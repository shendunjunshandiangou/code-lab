# 阶段 2 · 拆解：文章 → 原子知识点

## 输入 / 输出

- 输入：实例 `01_articles/` 中任务指定的文章
- 输出：实例 `02_atoms/` 中若干原子笔记文件

## 任务目标

把一篇文章拆解为多个**原子化知识点**（atomic notes），每个知识点一个独立文件。

## 原子合格三标准（逐条自检）

1. **可独立解释**：不依赖原文上下文也能讲清楚。
2. **可独立搜索**：这个文件名本身就是一个可以被检索、被引用的概念。
3. **可合并的不强拆**：两个概念总是同时出现且无法分开时，合并成一个原子。

## 文件命名规范

```text
领域-子领域-概念名.md
```

示例：`数据分析-SQL-窗口函数排名场景.md`、`AI编程-ClaudeCode-斜杠命令用法.md`

- 领域取自实例 `项目配置.md` 的主题领域
- 概念名要具体，避免"XX介绍""XX概述"这类空泛命名

## 每个原子的内容结构

```markdown
---
来源文章: "[[<文章文件名>]]"
标签: [<子领域>]
---

# <概念名>

## 一句话定义
<不超过两行>

## 详细说明
<从文章中提取的完整解释，可含表格/代码/步骤>

## 相关原子
- [[<相关原子1>]]
- [[<相关原子2>]]
```

## 执行规则

- 若 `01_articles/` 文章含 `## 文章正文` 节，**拆解范围止于该节之前**（含摘要与结构化板块）；`## 文章正文` 仅供人类通读，不生成新原子。
- 拆解前先通读全文（在上述范围内），列出候选知识点清单，再逐个生成文件。
- 若某原子文件已存在（其他视频拆过同一概念）：**不覆盖**，对比两者内容，把新增角度合并进已有文件，并在其"来源文章"里追加新来源。模式 B（主题搜索）下多个 UP 主讲同一概念是常态，观点冲突时在原子内并列呈现："A 的做法 / B 的做法"，不擅自裁决。
- 与主题领域无关的内容（闲聊、抽奖、恰饭）不生成原子。
