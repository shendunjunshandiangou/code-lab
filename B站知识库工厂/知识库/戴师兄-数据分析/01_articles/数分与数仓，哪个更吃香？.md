# 数分与数仓，哪个更吃香？

**摘要**：从薪资、入门难度、发展上限、工作体验四维度对比——数仓下限更高、入门更难、工作更枯燥；数分上限更高、体验更好、入门更易。长远看，分析师应懂数仓，否则搭指标体系需整队数仓。

## 一、薪资

| 维度 | 数分 | 数仓 |
| --- | --- | --- |
| 下限 | 较低（中小厂五六千也有） | 较高（需专业技术） |
| 上限 | 可转商分/战略/策略，吃项目红利 | 偏技术架构，上限因人而异 |
| 入行保障 | 追求可能性选数分 | 追求保底选数仓 |

## 二、入门难度

- **数分**：Excel/SQL/Tableau/Python，工具种类多但难度低；技能少可去中小厂
- **数仓**：Hadoop/Spark、调度、分层、维度建模、链表开发等，技术栈深；最快路径也需 MySQL + 大数据框架

## 三、发展前景

- 数仓技术方向：实时数仓降本增效；方法论成熟（如《数仓工具箱》），国内做不好常因**不懂业务、指标没梳理、不愿投入**
- 数分：吃经验，商分/策略好坏难量化；AI 与 Fabric 等会让数仓更简单，但业务知识与用户体验仍需人探索
- **对个人**：先学扎实再谈上限，3 年玩不转数仓、5 年玩不转商分都正常

## 四、工作体验

| 岗位 | 体验 |
| --- | --- |
| 数仓 | 建表写 SQL，枯燥；偏安逸但需持续优化表结构与效率；**几乎不直接接触业务** |
| 数分 | 也有取数报表，但能接分析需求、跟业务/产品/运营沟通，工作体验更好 |

## 五、长远建议

- **分析师必懂数仓**：懂则可招小朋友配合搭仓；不懂需整队数仓，管理压力大
- **数仓必懂分析**：不懂业务价值，需求永远做不完、面试说不出东西
- 数仓背景可先数仓再尝试分析；跳槽方向（数据工程/数分/商分/运营）别锁死技术路径

## 文章正文

数分和数仓到底选哪个？戴师兄从**薪资、入门难度、发展上限、工作体验**四方面盘点，并给出长远建议。

### 薪资

**数分下限更低、上限更高。** 数仓一开始就要学很多技术（至少 SQL、搭库）；数分要求参差不齐，很多中小企业五六千也招「分析师」做取数。爬取数据看，**数仓下限比数分高**。上限方面，分析师可转商分/战略/策略，**吃项目提成和红利**，上限比数仓高；若做到数仓架构负责人 vs 数分总监/业务负责人，则因人而异无法对比。入行要保障 → 数仓；追求后续可能性 → 数分。

### 入门难度

**数分入门更低**——Excel/SQL/Tableau/Python，会的少就去中小厂低要求岗位。**数仓**要 Hadoop/Spark、任务调度、分层逻辑、维度建模、链表开发等，考察更细；最快路径也需 MySQL + 大数据框架。数分工具种类可能更多但**难度低**；没数据基础、没编程的人学数仓「会很痛苦」，要先补信息工程技术，不像数分自学课相对简单。

### 发展前景

「数仓已死？」——技术上看还有**实时数仓**（降成本、提效率）等空间；1996 年的《数仓工具箱》方法论已很成熟。**国内做不好常因：不懂业务、指标没梳理好、穷不愿意投入**——天天省资源省出问题；企业愿意花时间花钱建数仓，数仓就没问题，数仓工作人员也会很舒适。

分析师侧较混乱——商分/策略好坏难量化，**吃经验**。ChatGPT、微软 Fabric 等会让数仓工作更简单高效，但分析仍有行业知识、业务经验、用户体验要探索；数仓技术层面突破有限，类似 Java 等偏编程领域。但对普通人：**先学出色再谈上限**——「3 年玩不转数仓、5 年玩不转商分都正常」；一开始别问上限高不高，先问努不努力。

### 工作体验

**数仓：干三个月就觉得「哇我就是建表」**——清理数据资产、写 SQL、建表，枯燥；往上做框架和数据治理要求极高。偏安逸又安逸不起来——需求要快、质量要高，表结构要持续优化，**没有尽头**。且**几乎不直接接触业务**，跟分析师沟通，分析师再跟业务沟通。

**数分：** 也有取数报表，但能接分析需求，跟业务/产品/运营/开发聊，还能做分析「花活」，**工作体验更好**。

### 长远建议

**分析师必懂数仓**——不懂则接新业务无法高效搭指标体系，要招整队数仓，管理压力巨大；懂则招一个小朋友配合即可。**数仓必懂分析**——不懂则需求永远做不完，不知道业务为什么要这个数、能发出什么价值，面试也说不出东西。

数仓开发相关专业可先数仓再尝试分析；未来做数据工程/数分/商分/运营看跳槽转机，**别把技术路径锁死**——路越宽、能力越稀缺。
