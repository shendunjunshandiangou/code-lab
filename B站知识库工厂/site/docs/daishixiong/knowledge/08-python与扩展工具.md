## 8.1 先定位：分析师到底该怎么用 Python

### 8.1.1 你是「领域用户」，不是程序员

**生活化类比**：会开车的人不需要会造发动机。程序员是造车的（语言创造者、工具创造者、应用开发者），分析师是开车送货的——你学驾驶技术（基础语法）和熟悉自己那条路线（领域包），就够了。

**技术定义**：按[数据分析-Python-领域用户定位](/daishixiong/atoms/数据分析-python-领域用户定位.html)，Python 使用者分四类：

| 类型 | 做什么 | 分析师是不是这类 |
| --- | --- | --- |
| ① 语言创造者 | 维护 Python 语言本身 | ❌ |
| ② 工具/包创造者 | 做 TensorFlow、Pandas、Django 等 | ❌ |
| ③ 应用开发者 | 用框架做知乎、B 站、Instagram 等应用 | ❌ |
| ④ **领域用户** | 用 Python 解决本行业重复劳动 | ✅ 分析师在这里 |

分析师属于第四类：非程序员，用 Python 自动化「**手动也能做**」的工作以提效。

三个推论直接决定你的学法：

- **手动做不出的，代码也做不出**。戴师兄原话：领域用户只是用 Python 做「手动也能做」的事——你想用 Excel 做两张表的笛卡尔积，手动几乎不可能，但 Python 可能两行解决；你想分析一个你根本不懂的业务问题，代码帮不了你。
- 核心价值是**效率**，不是开发产品。工具提效，但**增长靠人的策略**——工具不会自己启动，不会自己写代码，不能带来收入。
- **须先积累岗位/行业经验再学工具**（与 Excel/SQL/BI 同理）。没工作过的初学者，和有岗位经验的人学同一门课，感受完全不同——因为你不知道「手动做」长什么样。

**你可能会问**：招聘 JD 上写着「熟悉 Python 优先」，我不学是不是没竞争力？——注意是「优先」不是「必须」。对多数业务分析岗，Excel + SQL + 业务思维是硬门槛，Python 是加分项。它的正确位置是核心技能就位之后。

### 8.1.2 学什么：Pandas + 一个成型项目，仅此而已

**生活化类比**：健身房几十台器械，教练只让新手练三个基础动作。Python 有约 **60 万个**第三方包，分析师真正要精的只有一个。

**技术定义**：按[数据分析-Python-Pandas分析师定位](/daishixiong/atoms/数据分析-python-pandas分析师定位.html)，分析师的 Python 只需掌握 **Pandas**（Python 里处理表格数据的核心包，覆盖 90% 以上分析师的数据处理需求）+ 一个**成型项目**证明能力。

**必学清单**：

- 基础语法（变量、数字/字符串/字典/列表、if/for 循环、函数——领域用户「五个基础」就够）
- **Pandas**：`read_excel` / `read_csv` 读表、`head` 预览、`dtypes` 看字段类型、`concat` 合并、`to_excel` 导出；复杂场景再加 `groupby` 聚合、条件筛选
- **PyGWalker**：Python 内类 Tableau 的拖拽可视化（2023 版课程推荐，详见 8.3）

**环境搭建（课程实操路径）**：

1. 安装 **Anaconda**（百度搜索 Anaconda Download → 选 Graphical Installer；Windows 直接装，Mac 按芯片选）
2. 安装路径建议留足磁盘空间（可装 D 盘/E 盘）
3. 任意文件夹地址栏输入 `jupyter notebook` 回车 → 浏览器打开代码编辑器
4. 新建 `.ipynb` 文件即可写代码

**Jupyter 必会快捷键**（戴师兄课上教的三个就够）：

| 操作 | 快捷键 |
| --- | --- |
| 新增代码块 | 点击空白处 → 按 `B` |
| 删除代码块 | 选中块 → 连按两次 `D` |
| 运行当前块 | `Shift + Enter` |

课件本身也是 `.ipynb` 后缀——下载后双击就能复现视频里的代码。

**必做**：一个完整的「获取 → 清洗 → 分析 → 出图/出报告」**项目**，而不是小游戏或 demo——面试要的是这个。

**明确不做**：

- 日常可视化不用 Python 写（BI 拖拽更高效，见 8.3.1）
- 不深钻爬虫攻防（requests/scrapy 投入产出比低、有合规风险，见第四章三不原则；生产环境优先采集器/RPA）
- 不追求全包精通（NumPy、Sklearn 等按目标岗位 JD 缺什么补什么）
- Python 和 R 二选一时，新手选 Python

### 8.1.3 面试笔试怎么应对

**技术定义**：按[数据分析-Python-面试笔试应对](/daishixiong/atoms/数据分析-python-面试笔试应对.html)，Python 面试的底层逻辑是：企业雇人看的是**业务效率与增长**，工具服务于业务，不是为了刷题。戴师兄的暴论：企业录用永远看能否给业务带来增长、能否提升效率——你学的每个工具都要能回答「它服务什么业务环节」。

**最小可运行话术**（三个高频场景）：

1. **「你会 Python 吗？」** → 直接说会，然后反问：「具体是什么需求、团队用哪个包？」（60 万个包不可能全会；你没用过 B 包，不能否认你会用 A 包的事实）
2. **「简历上这段 Python 经历讲讲？」** → 如实讲用法，重点强调**效率提升**了多少（例如「手动合并 5 张表要半天，脚本 10 秒」；有增长数据更好）
3. **被问到陌生场景/没用过的包** → 承认没用过该包，但会 Python、新包上手快；再反问团队用什么包、是否要接手存量代码、还是希望你找更好用的方案

**笔试三类**：

| 类型 | 难度 | 应对 |
| --- | --- | --- |
| 邮件离线题 | 低 | 规定时间内写好发回；不会可以问 AI |
| 在线笔试 | 高 | 需要一定积累，但**不必海量刷题** |
| 线下笔试 | 高 | 同上 |

戴师兄在 AI 时代的建议：刷题投入产出比下降，**岗位与业务经验**比题量重要；但完全没有基础语法，在线笔试仍会吃亏。

**你可能会问**：说「会」再反问，算不算吹牛？——不算。领域用户本来就是按需查 API 干活的；关键是你确实具备「拿到新需求能快速上手」的能力，以及别在简历上写自己没做过的事。

## 8.2 场景一：爬虫与网络请求

第四章讲过：生产环境优先用采集器/RPA。那 Python 爬虫学来干嘛？——**学原理**。懂了网络请求的原理，你用任何采集工具都心里有数，面试聊起来也有底气。

### 8.2.1 网络请求五步法

**生活化类比**：发请求就像点外卖：选好店（URL）、说清要什么规格（参数）、报上你的收货信息（headers 身份标识）、下单（发起请求）、拆包装吃（解析响应）。

**技术定义**：按[数据分析-Python-网络请求五步法](/daishixiong/atoms/数据分析-python-网络请求五步法.html)，爬虫本质是**把网络请求代码化**：定需求 → 构建请求 → 发起 → 解析 → 存储。

一个请求由四要素构成：

- **方式**：GET（占 90% 以上）或 POST
- **内容**：URL 中到 `?` 为止的部分
- **参数**：分页、分区等（浏览器开发者工具「负载/Payload」面板可见）
- **身份**：请求标头 headers（含 Cookie、User-Agent）；缺了服务器会拒绝响应——这是失败最高频的原因

**抓包标准动作**（30 分钟爬虫课的核心技能）：

1. 浏览器按 `F12` 打开开发者工具
2. 切到 **Network（网络）** 面板
3. 刷新页面
4. `Ctrl+F` 搜关键词（如「list」「rank」）定位装着目标数据的那个请求包
5. 看四个区域：**标头**（URL、GET/POST、请求标头）→ **负载**（参数）→ **响应**（数据长什么样）

**最小可运行示例**：

```python
import requests
res = requests.get(url, headers=headers, params=params)
```

五步串起来：确定站点 → 从抓包复制四要素 → `requests` 发起 → 按响应类型解析（见 8.2.2）→ `pandas.DataFrame` → `to_excel()` 存盘。

**你可能会问**：headers 从哪来？——抓包时在开发者工具里那个请求的「标头/Headers」区域直接复制即可，它就是你浏览器访问时自带的身份信息。不要手写瞎猜 UA。

### 8.2.2 三种响应解析方案

**生活化类比**：外卖到手有三种包装：饭盒分格装好的（JSON，直接吃）、整桌摆盘的（HTML，得自己夹菜）、一锅炖的（纯文本，得自己捞）。

**技术定义**：按[数据分析-Python-三种响应解析方案](/daishixiong/atoms/数据分析-python-三种响应解析方案.html)，HTTP 响应常见三种形态，各配一把工具：

| 类型 | 识别特征 | 工具 | 后续处理 |
| --- | --- | --- | --- |
| JSON | 花括号 `{}` 结构 | `response.json()` | 按字段路径取值，如 `data["data"]["list"]` → DataFrame |
| HTML | 标签嵌套、表格在 tbody 里 | **BeautifulSoup** | 定位标签提取；可让 AI 写提取规则 |
| 纯文本 | 长字符串里夹着数据片段 | **`re` 正则** | 匹配起止位置 → 有时需 `json.loads` 转字典 → DataFrame |

**案例一：B 站每周必看（JSON，课程主案例）**

```python
import requests
import pandas as pd

url = "..."          # 抓包复制，到 ? 为止
params = {...}       # 从负载面板复制
headers = {...}      # 从请求标头复制，字典格式

res = requests.get(url, headers=headers, params=params)
data = res.json()["data"]["list"]   # 按实际 JSON 路径调整
df = pd.DataFrame(data)
df.to_excel("每周必看.xlsx", index=False)
```

大多数爬虫场景以 **JSON** 为主——现代站点前后端分离，数据接口直接返回 JSON。

**案例二：游戏排行榜（HTML）**

响应是 HTML 页面时，用 BeautifulSoup 按标签定位（如 `tbody` 里的 `tr`/`td`）。戴师兄建议：规则可以让 AI 写，你负责验证行数、字段对不对。

**案例三：纯文本嵌入（正则）**

部分站点返回一大段文本，数据藏在固定模式里——用 `re` 匹配起止，取出后再 `json.loads`。注意：有些站点必须在 headers 里带特定字段，才会返回 JSON/文本而不是 HTML 壳。

**导出统一姿势**：`pandas.DataFrame.to_excel()`。分析师场景下，最终产物往往是 Excel 或进 Pandas 继续清洗，而不是直接进生产数据库。

**你可能会问**：BeautifulSoup、正则要系统学吗？——不用。领域用户的姿势是遇到哪种响应查哪种解析，甚至让 AI 帮你写提取规则，你负责验证结果对不对、有没有漏页。

## 8.3 场景二：Python 数据处理与可视化看板

### 8.3.0 Pandas 最小可运行路径（读—看—合并—出）

在讲看板之前，先把 Pandas 这条主线走通——这是分析师 Python 的「主菜」。

**学 Pandas 的正确姿势**（戴师兄方法论）：不要一上来背所有 API。打开 **Pandas 官方 Quick Start**，把它当成「多功能文具盒」——先认识核心区域（DataFrame），再按需拿「铅笔」（read）、「橡皮」（筛选）、「胶水」（concat）。

**DataFrame 是什么**：Pandas 最核心的数据类型，有行有列——你可以把它理解成「Python 里的 Excel 表」。

**官方教程第一屏就会教你的五个动作**：

```python
import pandas as pd

df = pd.read_csv("titanic.csv")   # 或 read_excel("订单.xlsx")
df.head(8)                        # 看前 8 行——head = 头部
df.dtypes                         # 看各字段类型——排查日期被读成数字等坑
df.to_excel("导出.xlsx", index=False)  # 写回 Excel
```

戴师兄翻译法：`import` = 导入工具；`read_csv` / `read_excel` = 读入；`head` = 看头几行；`dtypes` = 看数据类型；`to_excel` = 写出。就算拼不出英文，跑一遍看结果就懂。

**练手项目：合并文件夹里多张 Excel（课程完整案例）**

业务场景：5 个门店各一张日报表，要合成一张总表。手动复制粘贴能做完，但 1000 张表就会累死——这正是 Python 的价值。

**阶段 1：先合并 2 张（理解 `concat`）**

```python
df1 = pd.read_excel("一店.xlsx")
df2 = pd.read_excel("二店.xlsx")
all_data = pd.concat([df1, df2])
all_data.to_excel("合并结果.xlsx", index=False)
```

**阶段 2：合并 5 张（复制 read + 往 concat 列表里加表）**

```python
df3 = pd.read_excel("三店.xlsx")
# ... df4, df5
all_data = pd.concat([df1, df2, df3, df4, df5])
```

**阶段 3：文件夹内所有 xlsx 自动合并（`for` 循环 + `os`）**

课程里的思路翻译：

1. `import os` → 用 `os.listdir(文件夹路径)` 列出所有文件名
2. `for file in ...` 逐个文件名循环
3. `if file.endswith(".xlsx")` 跳过非 Excel 文件
4. `os.path.join(文件夹, file)` 拼完整路径 → `pd.read_excel` 读取
5. 把每个 `df` `append` 进列表 → 最后 `pd.concat(列表)` 一次合并 → `to_excel` 导出

这就是分析师最典型的 Python 场景：**重复、规则明确、手动能做但太慢**——不是让你写爬虫攻防，也不是让你训练模型。

**可视化前的聚合与筛选（PyGWalker 章节的衔接）**

订单数据拖进 PyGWalker 之前，常要做两件事（课程口述）：

- 把「订单创建时间」按**天**粒度聚合，看成交金额趋势
- **筛选**掉「已取消」订单——否则 GMV 虚高

这些在 Pandas 里用 `groupby`、布尔索引都能做；具体语法查官方文档或问 AI，领域用户不必背全。

### 8.3.1 先学方法：主动学习选工具法

这一节的方法论比工具本身更值钱——它是你以后面对**任何**陌生技术领域的通用姿势。

**生活化类比**：被动学习像照着三年前的菜谱做菜，食材（版本）早变了；主动学习像先想好今晚想吃什么，再去市场看有什么新鲜食材，回来对着包装说明做。

**技术定义**：按[数据分析-Python-主动学习选工具法](/daishixiong/atoms/数据分析-python-主动学习选工具法.html)，在陌生技术领域应**需求驱动主动探索**，四步：

1. 明确需求（如「想要像 Tableau 一样拖拽、少写代码」）
2. 搜索并打开**官网**（比二手课程准确）
3. 看一句话介绍 + 示例图，判断是否匹配需求
4. 按官方安装指引和 Quick Start 动手实现

为什么不能被动跟教程：投入大无产出、教程版本过时、视野窄（比如做 Web 只知道 Flask/Django）。Python 生态约 60 万个第三方包，必须用需求筛选，不可能全学。

另一个前提共识——**BI 优先原则**：没有硬性工具要求时，**BI 拖拽永远比写代码快**。戴师兄原话：「假如没有硬性工具要求，一定是 BI 工具的效率更高。」所以「用 Python 搭看板」通常发生在：公司硬性要求 Python 交付、需要嵌入已有 Python 流程、或你要证明自主探索能力。

### 8.3.2 两条路线：PyGWalker 与 Shiny

**生活化类比**：PyGWalker 像单反相机——单张照片（单图）又快又好；Shiny 像装修房子——先画户型图（布局），再往每个房间摆家具（组件），能做出一整套仪表盘。

**技术定义**：按[数据分析-Python-PyGWalker与Shiny看板](/daishixiong/atoms/数据分析-python-pygwalker与shiny看板.html)，Python 搭看板两条路线：

| 工具 | 优势 | 局限 |
| --- | --- | --- |
| **PyGWalker** | 一行代码启动交互可视化；字段拖到 X/Y 轴即出图；可选 AI 作图 | **无法组合多图仪表盘** |
| **Shiny** | 官方组件齐全：`titlePanel`、`layout_columns`（分栏）、`value_box`（指标卡）、筛选器；`server` 端响应联动 | 需搭配 Plotly 等作图库；本质是「画框嵌画」的 Web 框架 |

**PyGWalker 最小可运行路径**：

```python
pip install pygwalker   # 安装后需 restart Jupyter kernel

import pandas as pd
import pygwalker as pyg

df = pd.read_excel("订单路径.xlsx")
pyg.walk(df)            # 一行启动拖拽界面
```

与 Tableau 的差异：度量要拖到 **X 轴 / Y 轴** 等轴位才显示，不是直接拖到「行/列」 shelf。

**AI 作图**（课程演示）：自然语言描述「用面积图展示不同期数成交金额随日期变化」→ PyGWalker 生成草图 → 筛选器部分可能需手动补。AI 功能需按官网指引安装扩展（有免费档）。

**报错怎么办**：主动搜索报错信息（如 widget 显示问题）→ 按论坛方案装依赖 → 重启 Jupyter。这是领域用户的常态，不是失败。

**Shiny 最小可运行路径（四步，正是 8.3.1 的实操示范）**：

1. **构思看板轮廓**：标题区、3 个 KPI 指标卡、1 个日期筛选器、2 张趋势图——不必画到像素级
2. **列组件清单**：对应 `titlePanel`、`value_box`、`dateInput`、Plotly 图表
3. **逛官方文档**：在 Components 分类里找示例语法，复制能跑的模板
4. **照示例改参数**：把示例 KPI 从 "Sales" 改成你的 "GMV"；不会改可以让 AI 兜底

**代码结构三段式**：

```text
导入库与数据
  ↓
ui：layout_columns 分栏 + value_box 指标卡 + 筛选器 + 图表占位
  ↓
server：响应筛选器日期 → 计算 GMV 等指标 → 返回 Plotly 图
```

这个看板的意义不在产物本身，而在**它是你第一次 0→1 主动探索官方文档完成的作品**——第二个看板你可以照着复制，第三个陌生工具你也不慌了。

**你可能会问**：公司已有 Tableau/Power BI，我还要会这个吗？——日常报表不需要（BI 优先原则）。但它证明两件事：你能在无人带教的情况下啃官方文档解决新问题；你的 Python 不止会跑通教程。这两点在面试里比「会做看板」值钱。
