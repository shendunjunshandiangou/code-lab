# 第三章 · SQL 取数：数据在数据库里，怎么拿出来

> 学到这里，你已经完成了第一章的认知准备，也在第二章把 Excel 练成了第一个动手技能：会匹配、会条件汇总、会用 Power Query 合并表、会用透视表出报表。但你可能已经隐约感觉到一个问题：**这一切的前提是数据已经躺在 Excel 文件里了**。真实公司里，数据存在数据库中——你得先把它"取"出来。这就是本章要解决的问题：**取数**。
>
> 好消息是，你在第二章的积累会直接变现：Power Query 的合并查询对应 SQL 的 JOIN，追加查询对应 UNION，SUMIF 的"条件 + 汇总"思想对应 WHERE + GROUP BY。戴师兄反复强调的学法就是——**每学一个 SQL 关键字，都想想它对应 Excel 的什么操作**。
>
> 本章路线：先建立认知（SQL 是什么、学到什么程度、怎么学），再看大厂取数的真实环境（数仓），最后攻克核心技术（连接、子查询、窗口函数）和求职笔试套路——**本章后半段会带着你完整走通多道校招真题的演算过程**，不是只列结论。

## 3.1 SQL 认知与定位

### 3.1.1 SQL 是什么：一大能力、两大属性

**生活化类比**：戴师兄的原版类比——数据库是一座山洞金库，里面堆满金银财宝（数据）；SQL 就是那句"芝麻开门"的咒语。咒语念对了，金库为你打开；念错一个字，门纹丝不动。

**技术定义**：SQL（Structured Query Language，结构化查询语言）是对数据库发号施令的标准化语言，核心能力就一条：**对数据库发号施令**——创建库、查询数据、优化查询。它有两大属性，详见 [数据分析-SQL-SQL是什么一大能力两大属性](/daishixiong/atoms/数据分析-sql-sql是什么一大能力两大属性.html)：

1. **结构化**：
   - 数据层面：只处理行列规整的结构化数据，不能处理合并单元格、图片这类东西；修改数据是整行整列操作；
   - 语法层面：语句顺序固定、有严格的运行顺序，SQL 是单向执行的语言。
2. **标准化**：语法由 ANSI（美国国家标准学会）统一制定，主流数据库都兼容。学会 SQL 等于学会了数据库界的"通用语"——去哪家公司、面对哪种数据库，咒语基本都通用（前提是你有访问权限）。

**最小可运行示例**——你的第一句"咒语"：

```sql
SELECT * FROM 订单表;
```

意思是：从"订单表"里把所有字段（`*`）查出来给我看。

**你可能会问**：SQL 算编程语言吗？会不会像学 Python 那样难？——它比通用编程语言简单得多：没有复杂的程序结构，核心就是七句话的排列组合（下一节），而且每句话都能对应到你已会的 Excel 操作。戴师兄还提醒：见过 400–500 行的超长 SQL，超过这个长度就该用临时表拆分子查询优化，否则查询太慢——这是工作场景的事，求职阶段先掌握七句话的组合即可。

### 3.1.2 七大核心语句与执行顺序

**生活化类比**：去餐厅点菜。你（人）先说"我要一份鱼香肉丝"（SELECT，先想要什么）；但厨房（数据库）的动作顺序是：先进冷库拿原料（FROM）→ 挑掉坏的（WHERE）→ 分锅炒（GROUP BY）→ 装盘摆好（SELECT）→ 按上菜顺序端出（ORDER BY）。人和厨房的顺序天然不同。

**技术定义**：SQL 查询由七大语句构成，书写顺序固定为：

```text
SELECT · FROM · WHERE · GROUP BY · HAVING · ORDER BY · LIMIT
```

而数据库的**执行顺序**是：

```text
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
```

书写顺序与执行顺序不同，且各主流数据库用法一致、顺序不能随意调换。详见 [数据分析-SQL-SQL七大核心语句与执行顺序](/daishixiong/atoms/数据分析-sql-sql七大核心语句与执行顺序.html)。

**最小可运行示例**（七句齐上阵，先看个脸熟，各语句后面会逐个用到）：

```sql
SELECT 部门, COUNT(*) AS 人数     -- 要什么字段
FROM 员工表                       -- 从哪张表取（最先执行）
WHERE 状态 = '在职'               -- 行级筛选
GROUP BY 部门                     -- 按部门分组
HAVING COUNT(*) >= 2              -- 对分组结果再筛选
ORDER BY 人数 DESC                -- 排序
LIMIT 10;                         -- 只要前 10 行
```

为什么人写 SELECT 在前？——**以终为始**：先想清楚要什么字段，再想从哪张表取。为什么数据库 FROM 先行？——必须先定位到表，后面的操作才有对象、才高效。另外记住 SQL 是**单向执行**：跑到后面的步骤不会回头重跑前面的步骤。

**Excel 迁移对照表**（入门课 P1 推荐用法）：

| SQL 关键字 | Excel 对应操作 | 一句话记忆 |
| --- | --- | --- |
| FROM | 打开/选中一张工作表 | 先确定数据在哪张表 |
| WHERE | 自动筛选 / 高级筛选 | 聚合**前**筛行 |
| GROUP BY | 透视表行字段 | 按分类汇总 |
| HAVING | 透视表值筛选（较少用） | 聚合**后**筛组 |
| SELECT | 选择要显示的列 | 决定输出什么 |
| ORDER BY | 排序 | 升序/降序 |
| LIMIT | 只取前 N 行 | Top N 的粗暴版 |
| DISTINCT | 删除重复项 | 去重 |
| JOIN | Power Query 合并查询 | 先拼表再取字段 |

**你可能会问**：WHERE 和 HAVING 都是筛选，差在哪？——看执行顺序：WHERE 在分组**前**筛原始行，HAVING 在分组**后**筛汇总结果。"只统计在职员工"用 WHERE；"只要人数 ≥ 2 的部门"用 HAVING。

### 3.1.3 数分的 SQL 要学到什么程度

**生活化类比**：考驾照。兴趣玩赛车不设上限；应付考试要练指定科目；日常上路其实就那几个动作——但科目考不过就没有上路资格。

**技术定义**：数据分析岗位的 SQL 掌握程度分三档，详见 [数据分析-SQL-数分SQL学到什么程度](/daishixiong/atoms/数据分析-sql-数分sql学到什么程度.html)：

| 档位 | 要求 |
| --- | --- |
| 兴趣驱动 | 不设上限，多学的都会用上 |
| **求职** | 语法（WHERE/HAVING/聚合/窗口函数）+ **套路题**（连续登录、Top N）+ 逻辑嵌套 |
| 工作 | 把业务口语翻译成 SQL；很多需求其实因数据不支持而做不了 |

**最小可运行示例**——用这三档给自己定位：如果你的目标是拿到数分 offer，那么达标线就是"求职档"：七大语句熟练 + 窗口函数会用 + 连续登录和 Top N 这类套路题能当场做出来。笔试的前几题考语法，后 4–5 题趋近套路题；面试时面试官会**随口出套路题**（比如只告诉你"用户 ID + 登录日期两列，算连续登录"），没有时间让你现场查资料——**听完你讲思路就知道你会不会**，甚至不需要你完整写出来。

**你可能会问**：题目做不出来，是不是我 SQL 没学好？——不一定。戴师兄给的关键判断：做不出题可能不是语法问题，而是**数据处理思路**缺失——换成 Excel 你也不会做。这个判断直接引出下一节的学习方法。

### 3.1.4 怎么学：输入输出刷题法

**生活化类比**：学游泳不能只看教学视频，得下水；但瞎扑腾也不行，得"看一个动作、练一个动作"。

**技术定义**：学 SQL 的输入（课程、文章、面经）和输出（做题）最终都落到**刷题**上；理解每个关键字时做 **Excel 迁移**——想清楚它对应 Excel 的什么操作。详见 [数据分析-SQL-SQL学习输入输出刷题法](/daishixiong/atoms/数据分析-sql-sql学习输入输出刷题法.html)。

学习路径三步：

1. **Excel 迁移**：每学一个 SQL 关键字，对应到 Excel 操作（WHERE ≈ 筛选，GROUP BY ≈ 透视表行字段，JOIN ≈ Power Query 合并查询……）；
2. **输入**：刷题、课程、文章、面经；
3. **输出**：继续刷题。

**最小可运行示例**——卡题时的自我诊断表：

| 症状 | 可能原因 | 对策 |
| --- | --- | --- |
| 完全没思路 | 数据处理流程没理解（Excel 也做不出） | 回头补数据处理思路 |
| 有思路但写不出 | 某个 SQL 语法点没掌握 | 定点补语法 |

检验标准：有了思路之后，**用 Excel 或 SQL 都能把它复现出来**，才算真正掌握。

**你可能会问**：有没有系统的入门课可以跟？——戴师兄的 SQL 免费入门课分三个分P：P1 基础查询（SELECT/FROM、WHERE、ORDER BY、LIMIT、DISTINCT、基础聚合），P2 多表进阶（JOIN、四种子查询、GROUP BY/HAVING），P3 窗口函数（ROW_NUMBER/RANK/DENSE_RANK、PARTITION BY），配合练习平台逐题实操。详见 [数据分析-SQL-SQL入门课程知识框架](/daishixiong/atoms/数据分析-sql-sql入门课程知识框架.html)。本章 3.3 节正是沿着这个框架展开的，并在 3.3.5 用真题把 P1–P3 串起来演算。

## 3.2 大厂取数与数仓

学语法之前，先看看大厂里"取数"这件事的真实样子——它可能和你想象的"分析师天天噼里啪啦写 SQL"很不一样。

### 3.2.1 大厂取数三级路径

**生活化类比**：家里想喝水：第一选择是拧开饮水机（现成的）；没有就烧壶开水（半自助）；实在不行才去打井（自己从头干）。

**技术定义**：大厂日常 80–90% 的取数需求**不需要手写 SQL**，取数遵循三级优先级，详见 [数据分析-SQL-大厂取数三级路径](/daishixiong/atoms/数据分析-sql-大厂取数三级路径.html)：

| 优先级 | 方式 | 覆盖场景 |
| --- | --- | --- |
| 1 | 看板直接下载 | 常用/核心指标已经固化在看板上 |
| 2 | 数据集可视化自助查询 | 看板没有、但数仓已加工好的数据 |
| 3 | 手写 SQL | 从应用层取数，通常只需要 WHERE 筛选 |

三级都覆盖不了？——提需求，让数仓团队加工新表、新指标。

**最小可运行示例**——老板要"上周华东区 GMV"时的标准动作：先找现有看板能不能直接下载；不行就去数据集平台拖拽查询；再不行才打开 SQL 编辑器，从应用层的表里 `SELECT ... WHERE 日期 BETWEEN ... AND 区域 = '华东'`。

戴师兄在直播里还补充了一个现实：有些公司运营手里就有数据、分析师根本碰不到 SQL——这很正常。但**你想往大厂走，SQL 缺不了**：不懂 SQL 就没法和数仓协作，笔试面试也过不了门。

**你可能会问**：既然 80–90% 不用写 SQL，那我还苦学它干嘛？——第一，剩下的 10–20% 恰恰是别人做不了、显你价值的部分；第二，和数仓团队协作、应急取数都要求你看得懂写得出；第三，别忘了 3.1.3——**求职笔试面试必考**，不会 SQL 你连门都进不去。

### 3.2.2 数仓四层分层：ODS-DWD-DWS-ADS

**生活化类比**：中央厨房。生鲜原样入库（ODS）→ 洗菜切配（DWD）→ 做成半成品料理包（DWS）→ 门店出餐给顾客（ADS）。每个环节只做一件事，哪个环节出问题一查便知。

**技术定义**：数据仓库（简称数仓，公司集中存储和加工数据的体系）普遍分四层，每层只做一件事，用存储空间换开发效率与问题定位能力。详见 [数据分析-SQL-数仓四层分层ODS-DWD-DWS-ADS](/daishixiong/atoms/数据分析-sql-数仓四层分层ods-dwd-dws-ads.html)。

| 层 | 职责 | 关键约束 |
| --- | --- | --- |
| ODS（原始层） | 业务系统数据原样同步 | **禁止加工**，与原始数据保持一致 |
| DWD（明细层） | 清洗：编码统一、UNION 等 | 只做清洗 |
| DWS（汇总层） | 宽表、指标汇总 | 基于 DWD 开发，像乐高组件 |
| ADS（应用层） | 面向业务的应用表 | **分析师主要取数层** |

为什么要分层？三个理由：

- **血缘追溯**：某张表数据报错，沿层级往上一层层定位问题源头；
- **避免循环引用**：没有规范时多表互相引用，一处崩全线崩；
- **复用组件**：下层拼好的"积木"，上层直接拿来用，不必重复开发。

**最小可运行示例**：层与层之间的数据流转靠 `INSERT INTO` / `INSERT OVERWRITE`（Hive 数据库常用写法）——从下层表取数、写入上层表。作为分析师，你日常取数认准 **ADS 应用层**即可。入职头晕的一个常见原因就是**不熟悉表环境**：别人问你问题，你不知道该从哪张表取、这个需求能不能做——所以读表名、理解分层，和学 JOIN 一样重要。

**你可能会问**：每家公司都是这四层吗？——不一定。有的公司还有 DWM、DM 等层，命名和层数各不相同。记住本质：分层是**人为制定的开发规范**，不是数据库的技术特性；理解"为什么分层"比背层名重要。小公司没有数仓团队时，规范由分析师自己定。

### 3.2.3 数仓表命名规范解读

**生活化类比**：快递单号。看似乱码，其实每一段都有含义——网点、线路、时效都编在里面。数仓表名同理，会读表名就等于拿到了"数据地图"。

**技术定义**：数仓表名由三部分组成：**层级前缀 + 业务词根 + 更新后缀**。看表名即可判断数据来自哪一层、什么业务主题、如何更新。详见 [数据分析-SQL-数仓表命名规范解读](/daishixiong/atoms/数据分析-sql-数仓表命名规范解读.html)。

1. **前缀**：ODS / DWD / DWS / ADS（哪一层）；
2. **词根词缀**：业务主题的简写（logistic→LGT、package→PKG），由数仓团队统一维护词根表；
3. **后缀**：频率 + 类型——D=天、W=周、H=小时；DF=全量（full）、DI=增量（incremental）。

**最小可运行示例**——读两个真实风格的表名：

- `DWS_bilibili_cos_EP_detail_DI`：DWS 汇总层 · B站课堂分批明细 · 按天**增量**更新；
- `ODS_bilibili_video_Reply_DF`：ODS 原始层 · B站视频评论列表 · 按天**全量**更新。

这套规范的目的是方便数据管理、治理和开发；大厂甚至会用 AST 解析（一种程序化的语法分析手段）自动检验表名是否合规、血缘是否合规。

**你可能会问**：我去的公司没有数仓、没有规范怎么办？——那就自己梳理业务过程后**自己定义规范**。小公司分析师往往身兼数据建设职责（第一章 1.4 提到过），这套命名思想直接可以搬过去用。

### 3.2.4 分区表与全量增量

**生活化类比**：档案室按年份分柜子存文件。你要 2023 年的合同，管理员直接开 2023 的柜子，不用把整个档案室翻一遍——这就是分区的意义。

**技术定义**：分区表在**逻辑上**是一个普通的筛选列，在**物理上**每个分区值对应一个文件夹。日期分区最常用；配合全量（DF）/增量（DI）后缀控制数据更新策略。详见 [数据分析-SQL-分区表与全量增量](/daishixiong/atoms/数据分析-sql-分区表与全量增量.html)。

常见两种分区：

1. **场景分区**：按平台分文件夹（抖音一个、B站一个）；
2. **日期分区**（最常用）：每天的新数据进当天的分区；取数时限制日期范围，可减少扫描量、加速查询。

全量与增量的区别：

| 类型 | 后缀 | 含义 |
| --- | --- | --- |
| 增量 | DI | 每天只把**新数据**放入新分区文件夹 |
| 全量 | DF | 每个分区存**截至当天的完整快照** |

**最小可运行示例**：取数时把分区列当普通字段用 WHERE 限制即可，完全不用关心底层文件夹：

```sql
SELECT * FROM ads_order_detail_di
WHERE dt BETWEEN '2026-07-01' AND '2026-07-07';
```

**你可能会问**：这个知识面试会考吗？——会。面试官常问"你用过 Hive 分区表吗"，答上"逻辑上是筛选列、物理上是文件夹、日期分区限制扫描范围加速查询"这几点，就能证明你摸过真实的大厂环境。

#### 工作场景对照：三级取数 vs 手写 SQL

戴师兄在「大厂 SQL 到底长啥样」里给了一个对照表，帮你判断什么时候该走哪条路：

| 老板说的话 | 你应该先做什么 | 手写 SQL 长什么样 |
| --- | --- | --- |
| "上周华东 GMV 多少" | ① 找 GMV 看板直接下载 | 通常不需要写 |
| "按渠道拆一下上周新增" | ② 数据集平台拖拽查询 | 通常不需要写 |
| "把高价值用户近 90 天行为拉出来" | ③ 打开 SQL 编辑器 | `SELECT ... FROM ads_xxx WHERE dt BETWEEN ... AND user_level = '高价值'` |
| "我想看用户点击路径和停留时长" | ④ 提需求给数仓 | 应用层没有，不是写 SQL 能解决的 |

分析师手写 SQL 的典型形态其实很简单——从 ADS 应用层取已经加工好的表，加上 `WHERE` 筛选日期、区域、用户群等维度，偶尔 `GROUP BY` 汇总。复杂的多表 JOIN 和窗口函数，更多出现在**笔试面试**和**数仓建设**里，而不是日常 80% 的取数场景。但正因为笔试必考，求职阶段必须把 3.3 节的套路练熟。

## 3.3 SQL 核心技术

这一节是本章的技术主体，对应入门课 P2、P3 的核心内容：连表、子查询、窗口函数。下面不只讲概念，还会用**校招真题完整演算**带你走一遍"读表 → 想思路 → 写 SQL → 排错"的全过程。

### 3.3.1 表连接原理与 LEFT JOIN

**生活化类比**：两份名单，一份有"姓名 + 电话"，一份有"姓名 + 地址"。Excel 里你可以用 VLOOKUP 把地址"劈"过来；SQL 里没有 VLOOKUP，你得先把两份名单**按姓名拼成一张大表**，再从大表里挑字段。

**技术定义**：SQL 中没有 VLOOKUP，跨表补字段必须用 JOIN（连接）——**先拼表，再取字段**。工作场景以 LEFT JOIN（左连接）为主：保留左表全部行，右表匹配不上的行以 NULL（空值）填充。详见 [数据分析-SQL-表连接原理与LEFT JOIN](/daishixiong/atoms/数据分析-sql-表连接原理与left-join.html)。

LEFT JOIN 的匹配过程（在脑子里放这部"慢动作"）：

1. 固定左表，逐行取出连接键（如用户 ID）；
2. 到右表逐行扫描找匹配；
3. 匹配成功 → 拼成新行；匹配失败 → 右表字段填 NULL；
4. 右表同一个键有多行 → 左表这一行会**膨胀成多行**（简单笛卡尔积）。例如左表 4 行、右表某键重复匹配，结果可能膨胀为 6 行。

**最小可运行示例**：

```sql
SELECT a.用户ID, a.订单金额, b.城市
FROM 订单表 a
LEFT JOIN 用户表 b
  ON a.用户ID = b.用户ID;
```

订单表的每一行都保留；某个用户在用户表里查无此人，他的"城市"列就是 NULL。

JOIN 选型建议：

| 类型 | 建议 |
| --- | --- |
| LEFT JOIN | **工作首选**，不丢左表数据 |
| INNER JOIN | 只保留两边都匹配的行，容易悄悄丢业务数据 |
| CROSS JOIN | 笛卡尔积，特殊场景才用 |

> 戴师兄提醒：不要偷懒直接写 `JOIN`（默认等于 INNER JOIN），显式写 `LEFT JOIN`，丢没丢数据自己心里有数。

#### 做题前提：先读表，再读题

戴师兄在真题课里反复强调：**做不出 SQL 题，大概率是表没看懂**。正确顺序是**先读表、后读题**——读题有时会误导你对表的理解；工作场景也一样，你应该先熟悉表环境，才知道老板的问题该从哪张表取数。

读每张表时，必须能回答：**这一行数据记录的是什么？** 注意"每一行"三个字——订单明细表的一行不一定是"一笔订单"（可能有子订单、负订单）。

#### 真题演算 1：会员等级访问统计（2023 携程校招）

**题目要求**：输出不同会员等级的访问人次总和、访问人数总和。

**第一步：读表**

| 表名 | 每一行记录什么 |
| --- | --- |
| 用户访问信息表 | 每一个用户**什么时候来、什么时候离开**（一人可能有多条访问记录） |
| 会员等级信息表 | **每一个用户**当前的会员等级和积分（一人一行） |

**第二步：想清缺什么字段**

- "会员等级"在会员等级表 → 必须 JOIN 补字段；
- "访问人次" = 数访问次数 → 用访问信息表 `COUNT(*)`；
- "访问人数" = 数来过多少人 → 对 `user_id` 做 `COUNT(DISTINCT user_id)`。

**第三步：写 SQL（手把手）**

```sql
SELECT
    b.vip_level AS 会员等级,
    COUNT(*) AS 访问人次,
    COUNT(DISTINCT a.user_id) AS 访问人数
FROM 用户访问信息表 a
LEFT JOIN 会员等级信息表 b
    ON a.user_id = b.user_id
GROUP BY b.vip_level
ORDER BY 访问人次 DESC;
```

**第四步：核对要点**

- 连接键是 `user_id`，不是别的 ID 字段（真题里常设陷阱）；
- 左表选访问信息表，因为人次统计需要明细行；
- `GROUP BY` 会员等级，`COUNT(*)` 数行数 = 人次，`DISTINCT` 数人去重。

这道题极其基础，但戴师兄讲它的目的就是让新手理解：**JOIN 就是把右表按连接键逐行匹配过来，补全你缺的字段**。

#### 真题演算 2：Tom 的 2023 年月度消费（2023 蚂蚁校招）

**题目要求**：查询客户 Tom 在 2023 年每个月的消费金额合计。

**第一步：读表**

| 表名 | 每一行记录什么 |
| --- | --- |
| 交易表 | 每一个客户在什么时候做了一笔什么类型的交易、金额多少 |
| 客户名称表 | 每一个客户叫什么名字（`cid` 对应客户 ID） |

**第二步：拆条件**

- 客户名 = Tom → 在客户名称表；
- 年份 = 2023 → 在交易表；
- 交易类型 = 消费（真题陷阱：`1` 表示消费，`0` 表示转账，漏筛就错）；
- 按月聚合金额 → `GROUP BY` 月份。

**第三步：写 SQL（手把手）**

```sql
SELECT
    SUBSTR(t1.trans_date, 1, 7) AS 月份,   -- 月是 7 位：2023-01
    SUM(t1.amount) AS 月消费金额
FROM 交易表 t1
LEFT JOIN 客户名称表 t2
    ON t1.cid = t2.cid          -- 注意是 cid，不是 tid
WHERE t2.cname = 'Tom'
  AND SUBSTR(t1.trans_date, 1, 4) = '2023'  -- 年是 4 位
  AND t1.t_type = 1            -- 1=消费，0=转账，易漏！
GROUP BY SUBSTR(t1.trans_date, 1, 7)
ORDER BY 月份;
```

**第四步：日期截取口诀**（戴师兄直播反复强调）

| 粒度 | `SUBSTR` 取几位 | 示例 |
| --- | --- | --- |
| 年 | 4 位 | `2023` |
| 月 | 7 位 | `2023-01` |
| 标准日期 | 10 位 | `2023-01-15` |

戴师兄在真题直播里还演示了**工作场景的简化写法**——知道 Tom 的 `cid=101` 时，甚至不需要 JOIN 客户名称表：

```sql
SELECT SUBSTR(trans_date, 1, 7) AS 月份, SUM(amount) AS 月消费金额
FROM 交易表
WHERE cid = 101
  AND SUBSTR(trans_date, 1, 4) = '2023'
  AND t_type = 1
GROUP BY SUBSTR(trans_date, 1, 7)
ORDER BY 月份;
```

笔试用 JOIN 是因为题目只给姓名不给 ID；工作中你早就知道用户 ID。两种写法结果一样，但考的是你能不能从题目条件反推出需要哪张表、哪个键。

**你可能会问**：做连表题总是无从下手？——做题前先做三件事：看懂每张表的**粒度**（一行代表什么）、有哪些**字段**、表和表之间**靠什么键关联**。这三件事没想清楚，写出来的 JOIN 必然错。这和 Power Query 合并查询前先确认匹配列是同一个道理。

### 3.3.2 子查询：WHERE IN 与 FROM 子查询

**生活化类比**：俄罗斯套娃。大娃娃（外层查询）肚子里装着小娃娃（内层查询）；要知道大娃娃长什么样，得先把小娃娃拿出来看完。

**技术定义**：子查询是嵌在 SQL 里的另一段 SQL，本质是**用内层 SQL 的结果替换占位**。执行原则：识别子查询 → **先执行最内层** → 用结果替换子查询文本 → 继续执行外层。详见 [数据分析-SQL-子查询WHERE IN与FROM子查询](/daishixiong/atoms/数据分析-sql-子查询where-in与from子查询.html)。

入门课框架里子查询有四种：WHERE IN、FROM 子查询、SELECT 子查询、相关子查询。前两种最常用：

#### 真题演算 3：WHERE IN 子查询（技术部 + 产品部员工）

**题目**：查询属于技术部或产品部的所有员工。

**写法**：

```sql
SELECT * FROM employee
WHERE dept_id IN (
    SELECT dept_id FROM department
    WHERE dept_name IN ('技术部', '产品部')
);
```

**执行过程（跟着动画走）**：

1. **最内层先跑**：`department` 表筛出技术部、产品部 → 得到 `dept_id` 列表，比如 `(101, 102)`；
2. **替换子查询**：外层变成 `WHERE dept_id IN (101, 102)`；
3. **外层再跑**：`employee` 表按 ID 列表筛人——符合的留下，不符合的"消失"。

本质就是**范围性查询**——用子查询框定 ID 范围，等价于你先查出一列 ID 再手动填进 IN 里。

#### 真题演算 4：FROM 子查询（员工数 ≥ 2 的部门）

**题目**：查询员工数不少于 2 人的部门全部信息。

**写法**：

```sql
SELECT * FROM department
WHERE dept_id IN (
    SELECT dept_id FROM (
        SELECT dept_id, COUNT(*) AS cnt
        FROM employee
        GROUP BY dept_id
    ) t
    WHERE cnt >= 2
);
```

**执行过程**：

1. **最内层 FROM 子查询**：`employee` 按 `dept_id` 聚合计数 → 生成临时表 `t`（部门 ID + 人数）；
2. **中间层**：从 `t` 筛 `cnt >= 2` 的部门 ID；
3. **外层**：`department` 表 WHERE 筛选，取出完整部门信息。

FROM 子查询的本质：**用一段 SQL 替代一张表**，后续当普通表继续筛选或连接。戴师兄提醒：动画只在学习阶段帮助理解，熟练后不需要在脑中模拟——但**读 SQL 永远从 FROM 开始**，先找数据来源，再逐层向外。

**你可能会问**：别人写的三四层嵌套 SQL，我从哪读起？——**永远从 FROM 开始读，逐层向外**。先找到最内层的 FROM 看它在准备什么表，再一层层看外面对这张表做了什么。读 SQL 的顺序就是数据库执行的顺序（3.1.2）。

#### 子查询后两种（入门课预告，待分P续拆）

[数据分析-SQL-SQL入门课程知识框架](/daishixiong/atoms/数据分析-sql-sql入门课程知识框架.html) 还列了 SELECT 子查询和相关子查询，本实例尚未逐题拆解，这里只列定义帮你建立索引：

- **SELECT 子查询**：在 SELECT 里嵌套标量查询，给每一行补一个计算字段（类似 Excel 在旁列写公式引用另一张表的单个值）；
- **相关子查询**：内层查询引用外层查询的字段，逐行关联计算（性能较慢，工作场景较少手写）。

求职笔试以前两种（WHERE IN、FROM 子查询）为主；遇到 SELECT/相关子查询题型，识别方式同样是**从最内层 FROM 读起**。建议后续补采入门课分P逐题任务后再扩写本节。

### 3.3.3 窗口函数与 Top N 套路

**生活化类比**：班里按成绩排名。GROUP BY 像只告诉你"全班平均分"（很多行汇总成一行）；窗口函数像**给每个学生在成绩单上标一个名次**——人还是那些人，行数不变，只是多了一列"排名"。

**技术定义**：窗口函数在已准备好的表上**按分区打标、排名，不改变行数**（对比 GROUP BY 会缩减行数）。Top N 类题目的通用套路是两步：先用窗口函数打排名，再在外层筛选排名。详见 [数据分析-SQL-窗口函数与Top N套路](/daishixiong/atoms/数据分析-sql-窗口函数与top-n套路.html)。

三种排名函数的差别在"同分怎么办"：

| 函数 | 同值行为 | 典型场景 |
| --- | --- | --- |
| `ROW_NUMBER()` | 强制给不同序号 | 每月 Top3 歌曲（同播放次用歌曲 ID 打破并列） |
| `RANK()` | 并列后跳号（1,1,3） | 竞赛排名 |
| `DENSE_RANK()` | 并列后不跳号（1,1,2） | 入职倒数第 N 名（同日期多人同排名，要全部取出） |

**核心口诀**：Top N 题 = **先打标，再取出排名第 N 的行**。

#### 真题演算 5：入职倒数第三名（牛客经典题）

**题目**：员工表有姓名、性别、生日、入职日期等字段，找出按入职日期升序排列的**倒数第三名**员工的全部信息。注意：同一入职日期可能有多人，要全部取出。

**为什么不用 `ORDER BY ... LIMIT 3`？**——同日期多人时，LIMIT 切不全；必须用窗口函数。

**思路**：入职日期倒序排名 → 取 `rk = 3`；同日期用 `DENSE_RANK` 给相同排名。

```sql
SELECT * FROM (
    SELECT *,
           DENSE_RANK() OVER (ORDER BY hire_date DESC) AS rk
    FROM employee
) t
WHERE rk = 3;
```

**演算过程**：

1. `DENSE_RANK() OVER (ORDER BY hire_date DESC)`：最晚入职的排第 1，同日期多人共享同一排名；
2. 外层 `WHERE rk = 3`：取出倒数第三名的所有员工（可能不止一人）；
3. 若题目要求同排名内再按员工编号排序，加 `ORDER BY emp_id`。

戴师兄直播总结：**Top N 所有题都这么做——先打标再取**，简单吗？能听懂说明语法基础过关；听不懂大概率窗口函数学得不够或这类题刷得少。

#### 真题演算 6：2022 年周杰伦歌曲每月播放 Top3（综合题）

**题目**（多表 + 多条件 + Top N）：流水表关联歌曲表、用户表；筛选年龄 18–25、年份 2022、歌手周杰伦；按月份 + 歌曲统计播放次数；每月取播放 Top3。

**思路拆解**：

1. 三表 JOIN：流水表 → 歌曲表（拿歌名、歌手）→ 用户表（拿年龄）；
2. WHERE 先筛：歌手 = 周杰伦、年份 = 2022、年龄 18–25（**先筛选再运算**，减少数据量）；
3. GROUP BY 月份 + 歌曲，`COUNT(*)` 得播放次数 `cnt`；
4. `ROW_NUMBER() OVER (PARTITION BY month ORDER BY cnt DESC, song_id)` 按月排名（同次数用 `song_id` 打破并列）；
5. 外层 `WHERE rk <= 3`。

```sql
SELECT month, song_name, cnt, rk
FROM (
    SELECT
        t1.month,
        t2.song_name,
        COUNT(*) AS cnt,
        ROW_NUMBER() OVER (
            PARTITION BY t1.month
            ORDER BY COUNT(*) DESC, t2.song_id
        ) AS rk
    FROM 播放流水表 t1
    JOIN 歌曲表 t2 ON t1.song_id = t2.song_id
    JOIN 用户表 t3 ON t1.user_id = t3.user_id
    WHERE t2.singer = '周杰伦'
      AND YEAR(t1.play_date) = 2022
      AND t3.age BETWEEN 18 AND 25
    GROUP BY t1.month, t2.song_name, t2.song_id
) ranked
WHERE rk <= 3;
```

**排错经验**（来自直播提交过程）：字段别名要和题目要求完全一致（如 `play_pv` 而非 `cnt`）；多表 JOIN 后 `SELECT *` 要标注来自哪张表（`t1.*` 还是 `t2.song_name`），否则字段来源混乱。

**你可能会问**：取前 3 名用 `ORDER BY cnt DESC LIMIT 3` 不是更简单吗？——两个坑：LIMIT 是对全表切前 3 行，做不到"每个月各取 3 个"；而且同值多人时（比如同一天入职好几个人），LIMIT 3 没法把并列者全取出来——这种场景要用 DENSE_RANK。

### 3.3.4 连续登录套路题

**生活化类比**：健身房打卡挑战"连续 7 天打卡领奖"。前台核对名单时怎么判断谁"连续"？把每个人的打卡日期排好队，找出中间没断的那几段。

**技术定义**：连续登录是 SQL 面试**最高频套路题之一**：给定用户 ID + 登录日期，判断哪些用户连续登录了 N 天。面试中常要求口述思路而非写完整代码。详见 [数据分析-SQL-连续登录套路题思路](/daishixiong/atoms/数据分析-sql-连续登录套路题思路.html)。

戴师兄在真题直播里直接说：**这道题可以背模板**——先排名、再日期减排名、再计数。没刷过这类题的人看答案一头雾水，但面试官很喜欢问，因为**听你讲完就知道刷没刷过题**。

#### 可背模板：日期减排名法（完整演算）

**题目变体**：注册表 + 登录表；求连续登录不少于 3 天的新注册用户（用户一天可能登录多次）。

**第一步：去重**——同一天登录多次只算一次：

```sql
SELECT DISTINCT
    user_id,
    DATE(login_time) AS login_dt
FROM 登录表;
```

**第二步：按用户分区、按日期排名**：

```sql
SELECT
    user_id,
    login_dt,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY login_dt) AS rk
FROM ( /* 上一步去重结果 */ ) t1;
```

排名效果示例（用户 1003）：

| login_dt | rk |
| --- | --- |
| 2月9日 | 1 |
| 2月10日 | 2 |
| 2月11日 | 3 |

**第三步：日期减排名 → 分组标记**

核心技巧：连续日期的"日期 − 行号"结果相同（等差数列公差相减）。

```sql
SELECT
    user_id,
    login_dt,
    rk,
    DATE_SUB(login_dt, INTERVAL rk DAY) AS tmp_dt  -- 连续段标记
FROM ( /* 上一步排名结果 */ ) t2;
```

用户 1003 前三天都算出来 `tmp_dt = 2月8日`（2月9日减1天、2月10日减2天、2月11日减3天，都是 2月8日）→ 说明这三天**连续**。如果 2月13日登录（中间断了 2月12日），减出来的 `tmp_dt` 就不再是 2月8日。

**第四步：按用户 + tmp_dt 聚合计数，筛 ≥ 3 天**：

```sql
SELECT user_id, tmp_dt, COUNT(*) AS 连续天数
FROM ( /* 上一步 */ ) t3
GROUP BY user_id, tmp_dt
HAVING COUNT(*) >= 3;
```

**第五步：变体处理**

- 题目要"最大连续天数" → 再套一层取 `MAX(连续天数)`；
- 题目要"新注册用户" → 加 `WHERE user_id IN (SELECT user_id FROM 注册表 WHERE ...)` 或 JOIN 注册表；
- 另一种思路是用 `LEAD` 判断相邻登录日期间隔是否恰好 1 天——逻辑等价，戴师兄说 LEAD 更通用，但日期减排名法更好背模板。

**面试口述版（四句话）**：

1. 按用户 + 日期去重、排序；
2. `ROW_NUMBER` 按用户分区给日期排名；
3. 日期减排名得连续段标记，同标记 = 同一段连续；
4. 按用户 + 标记聚合，筛 ≥ N 天。

**你可能会问**：面试时写不出完整代码会不会挂？——社招面试尤其看重的是**思路熟练度**：能把上面四步逐步讲清楚、说明每步为什么这么做，通常就达标了。它和 Top N 同属"没见过就很难临场想出来"的套路题——解法只有一个：提前刷到过。

### 3.3.5 从真题反推：SQL 做题标准流程

把 3.3.1–3.3.4 的真题演算浓缩成一张**可复用流程卡**，下次拿到任何 SQL 题先走这套：

| 步骤 | 做什么 | 对应本章例题 |
| --- | --- | --- |
| 1. 读表 | 每张表回答"一行是什么" | 访问表、交易表、登录表 |
| 2. 读题 | 拆出筛选条件 + 输出字段 + 聚合粒度 | Tom/2023/消费类型=1 |
| 3. 判缺字段 | 缺字段 → JOIN；缺中间结果 → 子查询 | 会员等级要 JOIN |
| 4. 写骨架 | FROM → WHERE → GROUP BY → SELECT | 月度消费聚合 |
| 5. 套路识别 | Top N → 窗口函数；连续 → 日期减排名 | 入职倒数第三、连续登录 |
| 6. 排错 | 检查连接键、漏筛条件、字段别名、日期截取位数 | `t_type=1` 易漏 |

## 3.4 求职笔试

### 3.4.1 笔试面试套路题方法论

**生活化类比**：考驾照科目二的倒车入库。第一次上车的人几乎不可能凭本能做对；练过标准流程的人闭着眼都行。差距不在天赋，在**见没见过**。

**技术定义**：SQL 笔试面试最主要考的是「套路题」——连续登录、Top N、在线人数、留存/复购率口述等固定模式的题目。没见过就做不出，刷够了才能举一反三。详见 [数据分析-SQL-笔试面试套路题方法论](/daishixiong/atoms/数据分析-sql-笔试面试套路题方法论.html)。

考察维度按重要性排序：

1. **套路题**（最主要）；
2. 语法（结合业务场景考，不会单独考"这个函数什么意思"）；
3. 逻辑嵌套（多层子查询）。

**最小可运行示例**——把本章内容组装成你的刷题清单：

| 套路类型 | 用到的本章武器 | 本章演算参考 |
| --- | --- | --- |
| Top N | 窗口函数三兄弟 + FROM 子查询（3.3.3） | 入职倒数第三（DENSE_RANK）、每月 Top3（ROW_NUMBER） |
| 连续登录 | 日期减排名 + 聚合（3.3.4） | 登录 ≥3 天新用户模板 |
| 多表聚合 | LEFT JOIN + GROUP BY（3.3.1） | 携程会员访问、蚂蚁 Tom 消费 |
| 子查询嵌套 | WHERE IN + FROM 子查询（3.3.2） | 技术部员工、≥2 人部门 |
| 在线人数 | 数据处理思路 + 聚合 | 原子仅有题型名，需补采真题 |
| 留存/复购率 | 常要求口述思路 | 原子仅有题型名，需补采真题 |

刷题时配合 3.1.4 的自我诊断：没思路 → 补数据处理流程；有思路写不出 → 定点补语法。戴师兄还提到美团等大厂笔试会考复购率口述——**说不出来就说明不熟练，甚至不需要你写代码**。

#### 笔试临场节奏（来自面经补充）

虽然「SQL 笔试临阵刷题法」原子挂在大纲第七章，但和本章直接相关，值得一并记住：

1. **考前 3 天**：每天 2–3 道套路题，重点练 Top N + 连续登录（本章 3.3.3–3.3.4 模板）；
2. **考场上先读表 2 分钟**：不要看到题目就写，先在本子上写"每张表一行是什么"；
3. **先写骨架再填细节**：`FROM → JOIN → WHERE → GROUP BY → SELECT`，最后套窗口函数；
4. **留 5 分钟检查**：连接键对不对、日期截取几位、有没有漏筛条件（蚂蚁 Tom 题的 `t_type=1` 就是典型漏筛）。

**你可能会问**：套路题做不出来，是不是我不适合干数据分析？——戴师兄的原话逻辑：**不是别人比你聪明，是你没见过这类题**。就像数学题，没刷过同类题纯靠现场想几乎不可能。学会套路之后，这些题会突然变得简单。

## 本章小结

把本章串成一条线：

1. **认知**：SQL 是对数据库发号施令的标准化语言（金库咒语），七大语句书写以终为始（SELECT 在前）、执行 FROM 先行；求职档要求 = 语法 + 套路题 + 逻辑嵌套；学法是 Excel 迁移 + 刷题输入输出。
2. **环境**：大厂 80–90% 取数靠看板和数据集，手写 SQL 主要发生在数仓 ADS 应用层；数仓四层（ODS 原样 → DWD 清洗 → DWS 宽表 → ADS 应用）是人为规范；表名三段式（层级 + 词根 + 频率/全量增量后缀）看名知义；分区表逻辑是筛选列、物理是文件夹。
3. **技术**：SQL 无 VLOOKUP，跨表必须 JOIN（工作首选 LEFT JOIN，警惕右表多行导致的膨胀）；子查询先内后外、读 SQL 从 FROM 读起；窗口函数打标不减行，Top N = 排名 + 外层筛选；连续登录 = 去重排序 + 日期减排名 + 聚合筛选。
4. **求职**：笔试面试的大头是套路题，刷够才能举一反三；本章 3.3.5 给出了从读表到排错的标准流程卡。

### 3.1 补充：P1 基础查询速通（入门课分P1）

虽然 [数据分析-SQL-SQL入门课程知识框架](/daishixiong/atoms/数据分析-sql-sql入门课程知识框架.html) 是框架级原子，但 P1 的 SELECT/FROM/WHERE 是后面所有真题的地基，这里用一道简单题把"P1 七句话"串起来——**对应入门课练习题的第一层难度**。

**题目**：从订单表查出 2023 年金额大于 1000 的订单，按金额降序，只显示前 5 条。

**读表**：订单表每一行 = 一笔订单（订单号、日期、金额、用户 ID……）。

**逐步写 SQL**：

```sql
-- 第 1 步：FROM —— 定位表（数据库最先执行）
SELECT order_id, amount, order_date    -- 第 5 步：SELECT —— 以终为始，先想好要什么
FROM orders                           -- 第 1 步：FROM
WHERE YEAR(order_date) = 2023         -- 第 2 步：WHERE —— 聚合前筛行
  AND amount > 1000                   -- 第 2 步：继续筛
ORDER BY amount DESC                  -- 第 6 步：排序
LIMIT 5;                              -- 第 7 步：限制行数
```

注意人写代码时 SELECT 写在最上面（以终为始），但数据库执行时 FROM 最先跑——3.1.2 的执行顺序表就是用来解决这个"脑内顺序 vs 机器顺序"的错位感的。P1 练熟后进入 P2 的 JOIN/子查询（3.3.1–3.3.2），再进入 P3 窗口函数（3.3.3），和戴师兄免费课的三段结构一一对应。

**练习平台建议**：戴师兄入门课配套练习题平台（视频简介/一元课群公告）按 P1→P2→P3 顺序刷；牛客网/LeetCode 数据库专题侧重套路题（Top N、连续登录）。两路并行——平台练语法手感，真题课练读表和思路。
