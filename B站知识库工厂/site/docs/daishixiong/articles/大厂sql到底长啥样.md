> **摘要**：揭开大厂 SQL 光环——日常取数 80–90% 靠看板和数据集，真写 SQL 多在数仓应用层。本课讲解大厂取数路径、数仓四层分层、表命名规范、分区表与全量/增量概念。

## 一、大厂日常怎么取数

**并非所有取数都要写 SQL。** 常用指标覆盖率 80–90% 不需要手写 SQL。

### 三级取数路径

| 优先级 | 方式 | 说明 |
| --- | --- | --- |
| 1 | **看板下载** | 常用/重要指标已固化在看板上，直接导出 |
| 2 | **数据集可视化查询** | 看板背后的数据集，类 Tableau 界面自助取数；数仓加工好后对运营等部门开放 |
| 3 | **写 SQL** | 看板和数据集都没有时，从数仓**应用层**（APP/ADM/ADS）取数，通常只需 WHERE 条件，无需大量加工 |

若应用层也没有 → 提需求让**数仓加工**。

## 二、数仓分层：为什么要分层

### 普遍四层

| 层级 | 全称 | 职责 |
| --- | --- | --- |
| ODS | 原始数据层 | 业务系统原封不动同步，**不做任何加工** |
| DWD | 明细层 | 数据清洗（编码统一、同类数据 UNION 等） |
| DWS | 服务层/宽表层 | 指标宽表、明细宽表，基于清洗后数据开发 |
| ADS | 应用层 | 面向业务的最终表，分析师主要取数层 |

### 分层意义

- **问题定位**：表报错可沿血缘往前追溯
- **避免循环引用**：无规范时「你用我的表、我用你的表」会导致一张表报错全部挂掉
- **乐高积木模式**：下层提供组件，上层拼装，不必每次从 0 到 1
- **代价**：用空间换效率，层越多存储越冗余

> 每家公司分层标准不同（还有 DWM、DM 等），取决于组织架构和数据情况。分层本质是**人为规范**。

### 怎么实现

通过 SQL 的 `INSERT INTO` / `INSERT OVERWRITE`（Hive 常用 overwrite）从下层取数写入上层。

## 三、表命名规范

通过表名即可判断表的内容和更新方式。示例解读：

- `DWS_bilibili_cos_EP_detail_DI`：DWS 层 · B站课堂 · 分批明细 · **按天增量**
- `ODS_bilibili_video_Reply_DF`：ODS 层 · B站视频评论列表 · **按天全量**

### 命名组成

1. **前缀**：层级（ODS/DWD/DWS/ADS）
2. **词根词缀**：业务主题简写（如 logistic→LGT，package→PKG），由数仓统一维护
3. **后缀**：
   - 更新频率：D=天、W=周、H=小时
   - 全量/增量：DF=全量（full）、DI=增量（incremental）

## 四、分区表

### 逻辑 vs 物理

- **逻辑上**：分区就是表里的一列，用法与普通字段筛选无异
- **物理上**：每个分区对应一个文件夹

### 常见场景

1. **场景分区**：按平台（抖音/B站/小红书）分文件夹，取数时 WHERE 限制单平台
2. **日期分区**（最常用）：每天新数据放入当天分区文件夹，取过去 7 天/1 个月时限制日期范围，**减少扫描量、加速查询**

### 全量 vs 增量

- **增量表（DI）**：新产生的数据放入新分区文件夹
- **全量表（DF）**：每次写入该日期分区的完整快照（如每天存一份截至当天的全量）

## 五、Hive / Spark SQL 与分析师

- 大厂内部多用本地数仓（数据保密），对外卖云服务
- 分析师需了解表从哪层来、怎么加工的，多了解大数据概念没有坏处
- 数仓分层本质是规范**表名、任务名、开发流程**——没有数仓团队时规范由自己定

## 文章正文

局长开场就点题：今天要揭开大厂 SQL 的「光环效应」——Hive、Spark SQL 听起来很高大上，但日常取数未必天天手写 SQL。直播最后还有彩蛋（录播会剪掉），到场观众能看到额外内容。

### 大厂到底怎么取数

**不是所有取数都必须写 SQL。** 常用、重要的指标，覆盖率大约 **80%～90%** 不需要你从零加工。

取数有三条优先级路径：

**第一，看板下载。** 常用指标都固化在看板上了，直接导出就行。局长现场演示：选中图表数据，下载打开就是数——「进字节只有第一天开心，离职也开心」，日常工作基本看板全覆盖，不会让分析师一个个去取。

**第二，数据集可视化查询。** 看板背后的数据集还在；不是所有内容都会做到看板上，数据集可以自助取数。界面像 Tableau：改年月、做折线图、看计费总金额趋势，开数据标签就能导出。数据集一般是数仓加工好后，对运营等其他部门开放——有合理正当需求，申请权限就能用。他吐槽某自研 BI「封神」双轴切换比较扯，但「比没有好」。

**第三，写 SQL。** 看板没有、数据集也没有，才真得上 SQL。但也不是从原始日志从头加工——数仓已经帮你做好**应用层**（各公司叫法不同：APP、ADM、ADS），你优先往最上层取，通常写一些 `WHERE` 条件就行，不需要大量加工。若应用层也没有，就提需求让数仓去加工。

### 数仓为什么要分层

大家面试、文章都听过 ODS/DWD/DWS/ADS 四层，但「为啥分层」往往讲不明白。普遍四层职责是：

- **ODS（原始数据层）**：业务系统原封不动同步，不做加工
- **DWD（明细层）**：清洗——编码统一、同类数据 UNION 等
- **DWS（服务层/宽表层）**：指标宽表、明细宽表
- **ADS（应用层）**：面向业务的最终表，分析师主要在这取数

分层意义局长用一张对比图讲清：左边有分层，某张表报错可以**沿血缘往前追溯**——是上面这张有问题，还是再往上四张有问题，定位非常清晰。右边没分层，「你用我的表、我用你的表」，循环引用，一张表报错全部挂掉。分层还是**乐高积木模式**：下层提供组件，上层拼装，不必每次从 0 到 1。代价是**用空间换效率**，层越多存储越冗余。

每家公司标准不同（还有 DWM、DM 等），取决于组织架构和数据情况——**分层本质是人为规范**。实现上通过 `INSERT INTO` / `INSERT OVERWRITE`（Hive 常用 overwrite）从下层写入上层。

### 表命名：看名字就知道表是什么

通过表名就能判断内容和更新方式。举例：

- `DWS_bilibili_cos_EP_detail_DI`：DWS 层 · B站课堂 · 分批明细 · **按天增量（DI）**
- `ODS_bilibili_video_Reply_DF`：ODS 层 · B站视频评论 · **按天全量（DF）**

命名三部分：**前缀**是层级；**词根词缀**是业务主题简写（如 logistic→LGT），由数仓统一维护；**后缀**是更新频率（D 天/W 周/H 小时）+ 全量 DF / 增量 DI。

### 分区表与全量/增量

**逻辑上**，分区就是表里的一列，用法跟普通字段筛选一样；**物理上**，每个分区对应一个文件夹。

两种常见场景：一是**场景分区**——按抖音/B站/小红书分文件夹，取数时 `WHERE` 限制单平台；二是**日期分区**（最常用）——每天新数据进当天分区，取过去 7 天或 1 个月时限制日期范围，**减少扫描量、加速查询**。

- **增量表（DI）**：新产生的数据放入新分区
- **全量表（DF）**：每次写入该日期分区的完整快照（比如每天存一份截至当天的全量）

### 分析师还要懂什么

大厂内部多用本地数仓（数据保密），对外卖云服务。分析师不必成为数仓工程师，但要了解表从哪层来、怎么加工的——多了解 Hive/Spark 等大数据概念没有坏处。没有专职数仓团队时，这套**表名、任务名、开发流程**的规范，也得自己定。
