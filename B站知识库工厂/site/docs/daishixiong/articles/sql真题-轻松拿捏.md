## 二、为什么需要表连接

**核心目的：原表字段不够用，需要从另一张表补充字段。**

与 Excel 的区别：

| Excel | SQL |
| --- | --- |
| VLOOKUP 直接「劈」字段过来 | 没有直接取字段的手段，必须**先把表拼在一起**，再取字段 |

所以叫「表连接」而非「字段连接」。

## 三、表连接过程（以 LEFT JOIN 为例）

两张表通过**连接键**（如 `dept_id`）匹配：

1. 固定左表，取左表第一行
2. 用连接键值去右表**逐行扫描**匹配
3. 匹配成功 → 拼成新行；匹配失败 → 右表字段为 NULL（LEFT JOIN 保留左表行）
4. 重复直到左表所有行处理完

### 数据膨胀

右表同一连接键有多行时，左表一行会匹配出多行——这叫**简单版笛卡尔积**。原本 4 行可能膨胀为 6 行。

### JOIN 类型速览

| 类型 | 工作场景 |
| --- | --- |
| **LEFT JOIN** | **最常用**，保留左表全部行，未匹配右表为 NULL |
| INNER JOIN | 只保留两边都匹配的行，丢数据不符合多数业务 |
| RIGHT JOIN | 固定右表找左表 |
| CROSS JOIN | 笛卡尔积，特殊场景使用 |

> 工作中推荐只用 LEFT JOIN，其他了解即可。直接写 `JOIN` 默认为 INNER JOIN，容易丢数据。

## 文章正文

SQL 真题系列第一课聚焦**表连接**。预计 3～4 节覆盖：表连接、子查询、窗口函数、综合题。戴师兄用自研动画演示 LEFT JOIN 匹配过程（约花 1.5 小时搓出），帮助理解「表连接不是字段连接」。

### 为什么需要表连接

**核心：原表字段不够，要从另一张表补字段。** Excel 用 VLOOKUP 直接「劈」字段；SQL 没有直接取字段手段，必须**先把表拼在一起再取字段**——所以叫表连接。

LEFT JOIN 过程：固定左表，用连接键逐行扫右表；匹配则拼行，不匹配则右表字段 NULL（保留左表行）。右表一对多会导致**数据膨胀**（简单版笛卡尔积）。工作中**推荐 LEFT JOIN**；直接写 `JOIN` 默认 INNER JOIN 易丢数据。

### 做题前提：先看懂表

做不出题大概率是**表没看懂**——每一行是什么粒度、有哪些字段、表间什么关系。例如「访问信息表」每一行 = 某用户某次来访的进入/离开时间；「会员等级表」每一行 = 某用户的等级与积分。

### 真题 1：23 年携程——会员等级访问人次/人数

会员等级来自等级表，访问人次/人数需 COUNT/COUNT DISTINCT `user_id`，用 `user_id` LEFT JOIN 两表后 `GROUP BY VIP 等级`。

### 真题 2：23 年蚂蚁——Tom 2023 年每月消费金额

交易表 + 客户名称表。工作场景可**投机取巧**：已知 Tom 的 `cid=101`，直接 `WHERE cid=101 AND 年份=2023 AND 交易类型=1（消费）`，不必 JOIN——「一看就知道是油头滑脑写 SQL 的老手」。若担心重名可用 `LIKE` 或 JOIN。时间筛选用 `SUBSTR(time,1,4)='2023'`，月粒度取 7 位；不建议 `LIKE` 模糊匹配时间（成本高）。

### 真题 3：23 年蚂蚁——还款能力级别逾期率

贷款表每一行 = **一笔合同**（非每个客户）；客户表为年龄与还款能力。求「有逾期行为客户的占比」→ 分子分母都是**人数**：合同表 `overdue_days IS NOT NULL` 时用 `COUNT(DISTINCT customer_id)` 作分子，分母为借款人数；粒度错把合同当客户会答错。百分比用 `CONCAT(ROUND(...)*100,'%')`。

### 其他要点

- 子查询 vs 表连接：Spark 里有时子查询更好，日常 LEFT JOIN 更直观。
- MySQL 是程序员常用库；分析师多用 Hive/Spark SQL，语法相近，先学 MySQL 即可。
- 分区表：大厂多为分区，查数习惯加 `LIMIT`；`INSERT OVERWRITE` 刷分区常见。
- 看板真实用途：**方便取数**（定时刷新、筛选），老板看的是报告不是看板。
