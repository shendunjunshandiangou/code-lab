# 数分与数仓，哪个更吃香？

<BiliPlayer :bvid="'BV1dj411e7tb'" />


**摘要**：从薪资、入门难度、发展上限、工作体验四维度对比——数仓下限更高、入门更难、工作更枯燥；数分上限更高、体验更好、入门更易。长远看，分析师应懂数仓，否则搭指标体系需整队数仓。

## 一、薪资

| 维度 | 数分 | 数仓 |
| --- | --- | --- |
| 下限 | 较低（中小厂五六千也有） | 较高（需专业技术） |
| 上限 | 可转商分/战略/策略，吃项目红利 | 偏技术架构，上限因人而异 |
| 入行保障 | 追求可能性选数分 | 追求保底选数仓 |

## 二、入门难度

- **数分**：Excel/SQL/Tableau/Python，工具种类多但难度低；技能少可去中小厂
- **数仓**：Hadoop/Spark、调度、分层、维度建模、链表开发等，技术栈深；最快路径也需 MySQL + 大数据框架

## 三、发展前景

- 数仓技术方向：实时数仓降本增效；方法论成熟（如《数仓工具箱》），国内做不好常因**不懂业务、指标没梳理、不愿投入**
- 数分：吃经验，商分/策略好坏难量化；AI 与 Fabric 等会让数仓更简单，但业务知识与用户体验仍需人探索
- **对个人**：先学扎实再谈上限，3 年玩不转数仓、5 年玩不转商分都正常

## 四、工作体验

| 岗位 | 体验 |
| --- | --- |
| 数仓 | 建表写 SQL，枯燥；偏安逸但需持续优化表结构与效率；**几乎不直接接触业务** |
| 数分 | 也有取数报表，但能接分析需求、跟业务/产品/运营沟通，工作体验更好 |

## 五、长远建议

- **分析师必懂数仓**：懂则可招小朋友配合搭仓；不懂需整队数仓，管理压力大
- **数仓必懂分析**：不懂业务价值，需求永远做不完、面试说不出东西
- 数仓背景可先数仓再尝试分析；跳槽方向（数据工程/数分/商分/运营）别锁死技术路径
